1 Resumen ejecutivo

Simulación de 25 usuarios médicos concurrentes ejecutando el flujo completo de generación de historia clínica vía texto, sobre la plataforma Scribe Nebula Medical. El objetivo es identificar el techo de rendimiento del stack antes de escalar la base de usuarios.

2 Objetivo del test

Se mide en esta iteración

  • Latencia end-to-end de generación (p50, p95, p99)
  • Tiempo al primer token de respuesta del LLM
  • Tasa de error del streaming SSE bajo carga
  • Comportamiento de los rate limiters in-memory bajo concurrencia
  • Uso de pool de conexiones Prisma/Postgres
  • LCP y Web Vitals de la UI en Chromium
  • Throughput real de Anthropic + OpenAI embeddings

Fuera de alcance (esta iteración)

  • Flujo de audio / transcripción Deepgram
  • Carga de adjuntos y RAG sobre documentos
  • Stress test de login SSO (auth.nebula.med)
  • Throughput de WebSocket / Deepgram live
  • Carga sobre entorno de producción (fase 2)
  • Mobile / Safari / Firefox rendering

3 Arquitectura del test

Cada VU corre un browser Chromium autónomo contra la aplicación. El runner k6 orquesta los 25 browsers y agrega las métricas.

4 Perfil de carga (stages)

El patrón simula la rampa de inicio de jornada médica: médicos que abren la aplicación escalonadamente, trabajan durante la hora pico y luego disminuyen la carga.

Fase Duración VUs inicio VUs fin Objetivo Rationale clínico
Warm-up 2 min 0 5 Calentar caché, pool DB, JIT del servidor Primeros médicos que abren la app
Ramp-up 3 min 5 25 Detectar cuello de botella en el ascenso Inicio de jornada matinal (08:00–08:30)
Sostenido 10 min 25 25 Medir estabilidad, memory leaks, pool exhaustion Hora pico de consultas (08:30–10:00)
Ramp-down 3 min 25 0 Verificar liberación de recursos, GC Cierre de jornada matinal

Justificación de 25 VUs: el rate limiter de inferencia AI es 30 req/min por user.id (in-memory, no Redis). Con 25 usuarios distintos el headroom teórico es 750 req/min. Una historia médica completa demora entre 20 y 90 s (prompt ~500 chars + streaming Claude), lo que genera 15–30 generaciones simultáneas en el pico — equivalente a una clínica con ~100 médicos activos en hora pico. El backend corre en una sola Azure Container Instance, por lo que 25 VUs es suficiente para encontrar el ceiling real del stack sin fragmentar el estado entre réplicas.

5 Flujo del usuario virtual (paso a paso)

Cada VU ejecuta el happy path completo de generación de historia médica mediante texto. No se usa audio en esta fase para evitar permisos de micrófono en Chromium y el rate limit de Deepgram (20 req/min por IP desde un runner único).

1
Autenticación — Fase 1 (dev-scribe)

El VU llama al endpoint de sesión CI que inyecta la cookie NextAuth directamente, evitando el SSO externo de auth.nebula.med. Requiere CI=true y E2E_TEST_USERNAME configurados en el contenedor dev.

GET /api/auth/ci-session
Set-Cookie: next-auth.session-token (JWT 24h)
2
Navegar a la pantalla de nueva consulta

El browser abre la ruta de nueva consulta. El servidor valida la sesión NextAuth y renderiza el shell.

page.goto('/chat/new')
GET /api/chat-rooms · GET /api/custom-templates · GET /api/user-encryption
3
Esperar hidratación completa de la UI

El logo del sidebar aparece con exactamente una instancia (el loader y el logo real coexisten brevemente). La presencia del logo indica que React terminó de hidratar y el estado del sidebar está estable.

expect(page.locator('svg[viewBox="0 0 100 108"].shrink-0')).toHaveCount(1)
4
Escribir la consulta médica

El VU rellena el textarea con un texto de ~500 caracteres que simula el dictado de un médico: anamnesis, síntomas, antecedentes y solicitud de historia clínica. El texto puede variar entre VUs para evitar caching agresivo.

page.getByPlaceholder('Empieza a escribir')

Selector alternativo si el placeholder cambia: [data-testid="chat-textarea"] o el role textarea dentro del form de chat.

5
Enviar el mensaje — disparo del endpoint crítico

Click en el botón de envío. El frontend abre una conexión SSE. El servidor inicia la cadena: rate limit check → RAG retrieval (Pinecone) → extracción de memorias (OpenAI) → streaming Claude → cifrado + persistencia (Postgres). Empiezan a contarse las métricas de latencia.

POST /api/chat-rooms/[id]/messages · Accept: text/event-stream
page.getByRole('button', { name: 'Enviar' })
6
Esperar el primer token y la respuesta completa

La URL del browser cambia de /chat/new a /chat/[id] una vez creada la sala. El VU espera a que aparezca el contenedor del mensaje del asistente con contenido mayor a cero caracteres. Se registran las métricas de tiempo-al-primer-token y tiempo-a-completado.

page.locator('[data-role="ASSISTANT"]').waitFor() page.locator('[data-role="ASSISTANT"]').textContent().length > 0
7
Verificar persistencia y registrar resultado

El VU valida que la historia médica generada esté disponible consultando el endpoint de mensajes. Registra el check como éxito o fallo para la tasa de éxito global.

GET /api/chat-rooms/[id]/messages
8
Pausa entre iteraciones

El VU duerme entre 10 y 30 segundos (distribución uniforme aleatoria) para simular el tiempo que un médico tarda en revisar la historia antes de la siguiente consulta. Esto evita que los 25 VUs golpeen el endpoint exactamente en sincronía.

sleep(randomIntBetween(10, 30))

6 Métricas y thresholds

Thresholds configurados directamente en el script k6. Si alguno se incumple, el test termina con código de salida distinto de 0 y bloquea un posible pipeline de CI.

Métrica Threshold Fuente Significado Criticidad
browser_web_vital_lcp < 3 s p95 Chromium Largest Contentful Paint — tiempo hasta que el canvas de chat es visible para el usuario Alto
browser_web_vital_cls < 0.1 p95 Chromium Layout shift — la UI de chat no debe repintarse caóticamente mientras llega el streaming Medio
http_req_duration{url:*/messages} < 40 s p95 k6 HTTP Duración total de la petición SSE de generación. Incluye TTFT + tiempo de streaming completo Crítico
time_to_first_token < 8 s p95 Custom trend Tiempo desde el POST hasta el primer chunk SSE. Mide latencia de RAG + inicio de inferencia Crítico
time_to_complete_generation < 35 s p95 Custom trend Tiempo desde el primer token hasta el evento SSE "done". Mide velocidad de tokens del LLM Alto
checks > 98% k6 checks Tasa de éxito de todos los assertions del flujo (hydration, mensaje ASSISTANT, persistencia) Crítico
browser_http_req_failed < 2% Chromium Requests HTTP fallidas dentro del browser (4xx, 5xx, network errors) Crítico
http_req_duration{url:*/ci-session} < 500 ms p95 k6 HTTP Endpoint de autenticación CI. Debe ser rápido para no contaminar el tiempo del flujo principal Medio

7 Prerequisitos críticos

Fase 1 — Recomendada
dev-scribe.nebula.med
  • Variable CI=true configurada en el contenedor dev (habilita /api/auth/ci-session)
  • Variable E2E_TEST_USERNAME con un usuario de prueba existente en la DB dev
  • k6 instalado: brew install k6 o binario de Grafana Labs
  • k6/browser habilitado: flag K6_BROWSER_ENABLED=true
  • Acceso de red desde el runner al subdominio dev (VPN si aplica)
  • Grafana Cloud o instancia local para recibir métricas vía K6_CLOUD_TOKEN
  • Coordinar ventana de prueba con el equipo dev (para no contaminar logs)
Fase 2 — Requiere aprobación
scribe.nebula.med (producción)
  • 25 cuentas loadtest-01..25@nebula.med pre-creadas vía SSO real
  • 25 archivos storageState-VU-XX.json generados con e2e/auth.setup.ts (válidos 24h por JWT NextAuth)
  • Script de regeneración automática de cookies ejecutado < 2h antes del test
  • Aprobación explícita del CTO / responsable técnico por escrito
  • Ventana fuera de horas clínicas (22:00–04:00 COT recomendado)
  • Banner de mantenimiento activo durante la ventana del test
  • Script de cleanup para eliminar historias médicas de test post-ejecución
  • Presupuesto LLM aprobado (estimar ~$15–40 USD por corrida completa)

Advertencia sobre rate limiters: los limiters de esta aplicación son in-memory (no Redis), lo que significa que no se comparten entre réplicas. Con la única instancia de Azure Container Instance actual, los 25 VUs con 25 user.id distintos disponen de 25 × 30 req/min = 750 req/min de headroom en el endpoint de inferencia. Cualquier escalado horizontal futuro cambiaría este comportamiento y requiere retestar.

8 Riesgos y mitigaciones

Riesgo Probabilidad Impacto Mitigación
Costo LLM inesperado (Anthropic / OpenAI) en prod Media Alto Estimar ceiling antes del test (~500 tokens/historia × 25 VUs × 10 iteraciones). Configurar alerta de gasto en el dashboard de Anthropic. Abortar si supera threshold. Preferir Fase 1 en dev donde los LLMs pueden apuntar a modelos más baratos.
Contaminación de datos médicos en prod (historias falsas) Alta Alto Usar cuentas loadtest-XX@nebula.med marcadas como de test. Ejecutar script de cleanup vía Prisma post-test que elimine todos los ChatRooms y mensajes de esas cuentas. Documentar el cleanup como obligatorio en el runbook.
Degradación de UX para usuarios reales durante el test Media Medio Ejecutar solo en ventana fuera de horas clínicas. Monitorear p95 de usuarios reales en Grafana en tiempo real. Botón de abort preparado. Comunicar la ventana al equipo de soporte.
Rate limit de terceros (Anthropic tier, Pinecone quota) Media Medio Correr Fase 1 en dev primero para calibrar throughput real. Verificar tier actual de Anthropic (tokens/min) y Pinecone (writes/s). Si el test detecta 429s, reducir VUs y extender el stage sostenido.
Cookies de sesión expiradas (JWT 24h) Alta Medio Para Fase 2: regenerar los 25 storageState.json < 2h antes de ejecutar el test. Automatizar con un script Playwright que haga login de cada cuenta secuencialmente. Añadir check de sesión activa al inicio del flujo del VU para fallar rápido.
Agotamiento del connection pool de Prisma (Postgres) Baja Alto Monitorear prisma_pool_connections_open durante el test. El pool default de Prisma es 10 conexiones; 25 VUs con operaciones concurrentes puede saturarlo. Si se detectan timeouts, considerar connection_limit=25 en el DATABASE_URL de dev.

9 Criterios go / no-go

El sistema se considera aprobado para 25 VUs si cumple todos los siguientes criterios al finalizar el stage sostenido:

GO — El sistema aguanta

  • checks rate > 98% al finalizar los 10 min sostenidos
  • time_to_first_token p95 < 8 s (el médico no percibe espera)
  • http_req_duration{messages} p95 < 40 s (historia completa)
  • Cero errores 5xx en el endpoint /api/chat-rooms/[id]/messages
  • browser_web_vital_lcp p95 < 3 s (UI carga rápido)
  • Sin memory leaks observables en el proceso Node.js durante el sostenido

NO-GO — Requiere acción

  • Cualquier threshold roto durante el stage sostenido
  • Errores 429 del endpoint de inferencia (> 1% de requests)
  • Degradación observada de latencia p50 > 2× respecto al warm-up
  • Pool de Postgres agotado (timeouts de conexión en logs)
  • Errores 500 en cualquier endpoint durante el pico
  • Cualquier dato de usuario real contaminado (en el caso de prod)

10 Roadmap — próximos pasos

Este documento de planificación es el paso 0. El roadmap de implementación es el siguiente:

Fase A
Script k6/browser
Crear loadtest/k6-generate-history.js implementando el flujo descrito en la sección 5. Incluir métricas custom Trend para TTFT y generación completa.
Fase B
Pipeline de sesiones
Script Playwright que genera y guarda 25 storageState.json para Fase 2 (prod). Scheduleable via CI para regenerar antes de cada corrida.
Fase C
Dashboard Grafana
Dashboard de Grafana Cloud con los paneles: VUs activos, p95/p99 de generación, TTFT, error rate, Web Vitals. Importable como JSON.
Fase D
Integración CI
Job de GitHub Actions que ejecuta el load test contra dev-scribe en cada release candidate. Falla el pipeline si algún threshold se incumple.
Fase E
Ejecución en prod
Una vez aprobados los resultados de dev, ejecutar contra prod en ventana nocturna con 25 cuentas de carga y cleanup post-test documentado.
